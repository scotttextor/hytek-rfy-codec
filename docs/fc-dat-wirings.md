was unwired** until this pass. The data was decrypted on 2026-05-01 (XOR

remaining "what does Detailer use here?" question.

## TRUSS GEOMETRY DATA ‚Äî field decode

The 53 rules are keyed by `GEOMETRY_<TYPE><PROFILE-INDEX>` where:

- **Type prefixes:**
  - `STDE`/`STDC` = STanDard truss (E=eaves/end, C=centre-hole/B2B)
  - `TRUE`/`TRUC` = TRUss (the principal truss type ‚Äî what TB2B uses)
  - `EAVE`/`EAVC` = EAVE truss
  - `RAFE`/`RAFC` = RAFter truss
  - `SADE`/`SADC` = SADdle truss
  - `NSTE` = Nested truss
  - `HALC` = HALf truss (Linear only)
- **Profile indices:**
  - 0 = 90mm/89mm normal-axis (`*E0`/`*C0`)
  - 1 = 75mm
  - 2 = 70mm normal-axis
  - 3 = 89mm + 70mm B2B normal (deep-axis sets)
  - 4 = 70mm using centre-web (B2B Deep Axis)
  - 5 = **89mm Linear Truss** (`construction_type=2`, special set)

**Key insight** ‚Äî `end_wb_setback` is **NOT 50mm for B2B as the brief assumed**.
It is 0 for all standard truss types. Only Linear Truss (TRUC5) uses 50mm.
The 50mm setbacks the brief saw in `simplify-tb2b-truss.ts` (`boxA + 50`,
`boxB - 50`) are likely from `shorten_dbles_wb=50` (the chord-doubles
near-end web suppression), not `end_wb_setback`.

## Ranked wirings by impact √ó confidence √ó ease

### Tier 1 ‚Äî HIGH confidence, narrow scope

#### W1. `cen_cen_wb_hole` = 120 ‚Üí wire as TB2B web-hole min separation

- **Field:** `TRUSS GEOMETRY DATA.GEOMETRY_TRUC{0..5}.cen_cen_wb_hole = 120`
- **Codec target:** `simplify-tb2b-truss.ts` web-hole emission. Currently the
  TB2B simplifier picks centerline-crossings without a min-separation guard
  on each chord. The `120` value defines that two web-holes on the same
  chord cannot be closer than 120mm centre-to-centre (in 89/90/75/70mm
  profiles ‚Äî same value across all C-types). Apply the constraint when
  multiple webs cross the same chord within 120mm ‚Äî only emit the closer-
  to-panel-point one.
- **Profile applicability:** Same `120` for all C-types. **No-op for current
  HYTEK setups** (they all use 120). Future-proofs for setups where the
  field differs. Also reasonably small enough to commit defensively with
  zero parity risk.
- **Confidence:** HIGH ‚Äî field name and value match the constraint we'd
  expect to see; cross-profile consistency removes profile-branching
  complexity.
- **Risk:** If we add the constraint and Detailer DOESN'T enforce it, we'd
  drop legit web-holes. Need to verify against corpus before landing ‚Äî only
  wire as defensive constant; don't activate the suppression unless we see
  evidence Detailer uses it.

#### W2. `end_wb_setback` = 50 (LINEAR only) ‚Üí already implicit but document

- **Field:** `GEOMETRY_TRUC5.end_wb_setback = 50`, all others = 0
- **Codec target:** Linear truss handling in `simplify-linear-truss.ts`.
  The Linear simplifier already doesn't emit web-holes near chord ends;
  this field documents the source.
- **Action:** No code change. Add comment in `simplify-linear-truss.ts`
  citing `GEOMETRY_TRUC5.end_wb_setback`.

#### W3. Wire MachineSetup ‚Üí FC_DAT consumer module

- Build `src/fc-dat-rules.ts` exposing typed accessors:
  - `getTrussGeometry(profile: number, type: 'STD'|'TRU'|'EAV'|'RAF'|'SAD'|'NST', deepAxis: boolean)`
  - Return `TrussGeometryRule` with the named fields above.
- Loaded once from `scripts/fc-dat-parsed.json` (or copy into `src/fc-dat.ts`
  as a static export to avoid runtime fs).
- Future wirings consume this layer instead of hardcoding.

### Tier 2 ‚Äî MEDIUM confidence, defensive

#### W4. `shorten_dbles_wb` (= 50) ‚Üí match TB2B `boxA + 50` / `boxB - 50`

- **Field:** `GEOMETRY_TRUC{0..4}.shorten_dbles_wb = 50` (Linear = ‚àí50)
- **Codec target:** `simplify-tb2b-truss.ts:494-495` ‚Äî `startPos = boxA + 50;
  endPos = boxB - 50;` (box-piece InnerDimple range).
- **Confidence:** MEDIUM ‚Äî value matches, but it's not strictly clear that
  this codec's `boxA + 50` is the same Detailer rule. Could also be from
  `setup.boxedFirstDimpleOffset` (= 50 for 70mm setup, 10 for 89mm ‚Äî but
  that's per-machine not per-truss).
- **Action:** Document the `50` literal with a citation, but don't yet swap
  it out ‚Äî the per-stick mapping isn't 1:1 (the codec uses one `50` for
  both ends, while .dat splits `shorten_dbles_wb` from `end_wb_setback`).

#### W5. WBUILD `"300 450"` service heights (already documented)

- **Field:** `WALL DEFAULT DATA.WBUILD.f12 = "300 450"`
- **Codec target:** Was the static `@296/@446` rule in `rules/table.ts`
  (commented as `300 - 4mm trim` and `450 - 4mm trim`). Now superseded by
  `simplify-wall-service.ts` dynamic z-line projection from XML.
- **Action:** None. The static rule is post-stripped by the dynamic
  simplifier; the source data is cited only for historical context.

### Tier 3 ‚Äî LOW immediate impact, future-proofing

#### W6. `web_increments` field ‚Äî per-profile

- **Field:** TRUC0=40, TRUC1=35, TRUC2=35, TRUC3=30, TRUC4=12, TRUC5=30
- **Codec target:** Web-spacing rounding in `simplify-tb2b-truss.ts`?
  Not currently used. Detailer rounds web positions to multiples of this
  increment. The codec could add `roundToIncrement(pos, increment)` for
  panel-point positions ‚Äî but the codec already gets exact world-coords
  from XML so this rule is likely redundant.
- **Action:** Document only. Don't wire ‚Äî the XML inputs already carry
  exact positions.

#### W7. `bc_dist_cen_hole` / `tc_dist_cen_hole` ‚Äî per-profile chord end offset

- **Field:** Profile-specific (61.5 for 89/90, 54.5 for 75, 50 for 70,
  44.5 for B2B 89, 35 for B2B 70, 44.5 for Linear).
- **Codec target:** Chord centre-hole emission near ends ‚Äî the FIRST
  centre-hole on a chord is offset by this distance from each end.
- **Confidence:** MEDIUM. The codec doesn't currently emit centre-holes
  on chords explicitly (TB2B emits Web@pt at crossings, NOT centre-holes
  along chord runs). So this likely applies to non-TB2B truss types
  (STDC = standard chord-truss) which the codec doesn't yet implement.
- **Action:** Document for future non-TB2B truss support.

## Implementation plan

Given the audit:

- **Highest-confidence wiring is W3** (build the consumer module) +
  documentation of W1/W2 ‚Äî but actually CHANGING parity numbers requires
  evidence that the codec is currently miscalibrated where the .dat data
  is right. After Agents O/V/X/Y/Z, the largest hardcoded constants have
  been wired to `.sups`; the .dat data largely overlaps (or are different
  axes ‚Äî e.g. `cen_cen_wb_hole` is a CONSTRAINT, not an offset).

- **The `300 450` lead I was given isn't a wiring target** ‚Äî that data is
  already overridden by per-job XML in `simplify-wall-service.ts`. The
  `4mm trim` in `@296/@446` was indeed `300 - 4` and `450 - 4`, but the
  whole rule is now post-stripped before serialisation.

- **The `50mm end_wb_setback` lead was misread by the brief** ‚Äî that field
  is 0 for standard trusses and 50 only for Linear Truss. The TB2B `+50/-50`
  in `simplify-tb2b-truss.ts:494-495` is at best `shorten_dbles_wb` (also 50,
  also profile-coincident) but the semantics are different (chord-doubles
  setback vs box-piece InnerDimple range).

- **`120` cen_cen_wb_hole** ‚Äî this IS new data. But to wire it would
  require modifying `simplify-tb2b-truss.ts` to enforce a 120mm web-hole
  min separation. Without evidence Detailer enforces this on the captured
  corpora, adding the constraint risks regression.

Net: this audit's primary value is to surface that the .dat / .sups data
mostly RECONFIRMS what Agents O‚ÄìZ already empirically derived from corpus.
The remaining 15‚Äì20% parity gap (HG260001 84.71%, HG260044 83.44%, HG260023
80.33%) is dominated by clusters that the .dat schema doesn't directly
address (apex-pair offsets, box-piece cap-stack patterns, frame-type-
specific cap stacks).

The honest path forward is W3 (consumer module + parsed JSON exports) so
future wirings have a clean foundation, plus documentation citations on the
existing literals. Aggressive parity pushes from .dat alone are unlikely.
